## 1. Executive Summary

This project implements an **automated contract testing framework** for microservices-based REST APIs. It ensures API compatibility between services at build time, preventing breaking changes from reaching production environments.

The framework uses **Spring Cloud Contract** to define, verify, and enforce API agreements between a Provider service and its Consumer(s). An AI-powered agent (Phase 7) will further automate contract generation by reading OpenAPI specifications.

**Key Outcomes:**
- Breaking API changes are detected automatically during CI builds
- Consumer and Provider teams stay in sync without manual coordination
- Contract verification is integrated into the standard Maven build lifecycle
- WireMock stubs enable Consumer testing without a running Provider

## 2. Problem Statement

### 2.1 Background

In a microservices architecture, services communicate over HTTP APIs. Each service is developed, tested, and deployed independently. This creates implicit dependencies between services — when one service changes its API contract, dependent services may break.

### 2.2 Identified Challenges

| # | Challenge | Business Impact | Risk Level |
|---|-----------|-----------------|------------|
| 1 | No formal API agreement between service teams | Uncoordinated changes cause integration failures | High |
| 2 | API changes are not communicated to consumers | Silent breakage in production | Critical |
| 3 | Integration testing requires all services running simultaneously | Expensive, slow, often skipped | High |
| 4 | Manual contract maintenance is labor-intensive | Contracts become stale and unreliable | Medium |
| 5 | No automated contract verification in CI/CD | Breaking changes pass through deployment gates | Critical |

### 2.3 Target Scenario

**Without contract testing:**
1. Team A (Provider) renames a JSON field `name` → `fullName`
2. Team A's unit tests pass — their code is internally consistent
3. Team A deploys to production
4. Team B's (Consumer) service expects `name` — **production incident occurs**

**With contract testing:**
1. Team A renames `name` → `fullName`
2. Contract tests fail at build time — **deployment is blocked**
3. Team A is notified of the contract violation before merge
4. Issue is resolved during development, not in production

### 3.1 High-Level Architecture

```
┌──────────────────────────────────────────────────────────────────┐
│                       CI/CD PIPELINE                             │
│   Build → Contract Verification → Report → Deploy (gated)       │
└──────────────┬───────────────────────────────┬───────────────────┘
               │                               │
  ┌────────────▼─────────────┐    ┌────────────▼─────────────┐
  │  PROVIDER SERVICE        │    │  CONTRACT DEFINITIONS    │
  │  (User Service :8080)    │    │  (YAML files)            │
  │                          │    │                          │
  │  Spring Boot REST API    │    │  Request/Response pairs  │
  │  OpenAPI spec generation │    │  Verified by SCC         │
  │  SCC Verifier plugin     │    │  Auto-generated by AI    │
  └────────────┬─────────────┘    └────────────▲─────────────┘
               │                               │
  ┌────────────▼─────────────┐    ┌────────────┴─────────────┐
  │  CONSUMER SERVICE        │    │  AI AGENT (Python)       │
  │  (Order Service :8081)   │    │                          │
  │                          │    │  OpenAPI spec reader     │
  │  Calls Provider via HTTP │    │  Contract generator      │
  │  Tests via WireMock stubs│    │  Drift detector          │
  │  SCC Stub Runner         │    │  Remediation advisor     │
  └──────────────────────────┘    └──────────────────────────┘
```

### 3.2 Contract Testing Flow

```
CONTRACT YAML FILES (shared source of truth)
         │
         ├────────────────────────────────────────┐
         ▼                                        ▼
   PROVIDER SIDE                           CONSUMER SIDE
   ─────────────                           ─────────────
   SCC Verifier reads contracts            SCC Stub Runner loads stubs
         │                                        │
   Auto-generates JUnit tests              Starts WireMock server
         │                                        │
   Executes against real API               Consumer tests against stubs
         │                                        │
   Pass → Generates stubs JAR ──────────► Stubs JAR (WireMock mappings)
   Fail → Build blocked
```

### 3.3 Inter-Service Communication

```
  Order Service (Consumer)                     User Service (Provider)
         │                                            │
         │  HTTP GET /api/users/{userId}              │
         ├───────────────────────────────────────────►│
         │                                            │
         │  200 OK { id, name, email, role }          │
         │◄───────────────────────────────────────────┤
         │                                            │
    OrderResponse = Order data + User data
```

## 6. Phase Details

### 6.1 Phase 1 — Problem Statement & Requirements Analysis

**Objective:** Define the problem space and establish success criteria for the contract testing solution.

**Findings:**
- Microservice teams lack a formal mechanism to agree on API shapes
- Integration test suites are brittle and expensive to run
- API changes propagate silently, causing production incidents
- Industry solution: **Consumer-Driven Contract Testing** (CDC)

**Success Criteria Defined:**
1. Breaking API changes must be detected at build time, not in production
2. Consumer tests must run without requiring a live Provider
3. Contract verification must integrate into standard Maven build lifecycle
4. The solution must support automated contract generation from API specs

---

### 6.2 Phase 2 — Technology Research

**Objective:** Evaluate and select technologies for contract testing, API documentation, and automation.

**Technologies Evaluated:**

| Technology | Decision | Rationale |
|-----------|----------|-----------|
| Spring Cloud Contract | **Selected** | Native Spring Boot integration, Maven plugin, YAML contracts, auto-generated tests |
| Pact | Evaluated | Strong multi-language support but less native Spring integration |
| OpenAPI / springdoc | **Selected** | Auto-generates API spec from code annotations; machine-readable |
| REST Assured | **Selected** | Fluent API testing DSL, first-class SCC support via MockMvc module |
| WireMock | **Selected** | Industry-standard HTTP mock server; SCC generates compatible stubs automatically |

**Key Architecture Decisions:**
- Contracts stored on **Provider side** (`src/test/resources/contracts/`) — Provider is the source of truth
- **YAML format** for contracts — human-readable, version-controllable
- **Stubs JAR** generated during Provider build and consumed by Consumer via local Maven repository
- **Matchers** (regex-based) used instead of exact values for resilient validation

### 6.3 Phase 3 — Provider API (User Service)

**Objective:** Build the Provider microservice that other services depend on.

**Service Details:**

| Property | Value |
|----------|-------|
| Service Name | User Service (Provider) |
| Port | 8080 |
| Base Path | `/api/users` |
| Storage | In-memory (ConcurrentHashMap) |
| Documentation | Auto-generated at `/v3/api-docs` and `/swagger-ui.html` |

**REST API Endpoints:**

| Method | Endpoint | Description | Response Codes |
|--------|----------|-------------|----------------|
| GET | `/api/users` | Retrieve all users | 200 |
| GET | `/api/users/{id}` | Retrieve user by ID | 200, 404 |
| POST | `/api/users` | Create a new user | 201, 400 |
| PUT | `/api/users/{id}` | Update an existing user | 200, 400, 404 |
| DELETE | `/api/users/{id}` | Delete a user | 204, 404 |

**Data Model (`User`):**

| Field | Type | Validation | Description |
|-------|------|-----------|-------------|
| `id` | Long | Auto-generated | Unique identifier |
| `name` | String | `@NotBlank`, `@Size(2,100)` | User's full name |
| `email` | String | `@NotBlank`, `@Email` | Email address |
| `role` | String | `@NotBlank` | Role (ADMIN, USER, MANAGER) |

**Error Handling:**
- Validation errors (400): Returns field-level error messages via `@RestControllerAdvice`
- Not found (404): Standardized error response with timestamp
- Server errors (500): Generic error response

### 6.4 Phase 4 — Consumer API (Order Service)

**Objective:** Build a Consumer microservice that depends on the Provider API, establishing the inter-service contract dependency.

**Service Details:**

| Property | Value |
|----------|-------|
| Service Name | Order Service (Consumer) |
| Port | 8081 |
| Base Path | `/api/orders` |
| Provider Dependency | Calls `GET /api/users/{id}` on Provider via `RestTemplate` |
| Configuration | `provider.api.base-url=http://localhost:8080` |

**REST API Endpoints:**

| Method | Endpoint | Description | Response Codes |
|--------|----------|-------------|----------------|
| GET | `/api/orders` | Retrieve all orders (enriched with user data) | 200 |
| GET | `/api/orders/{id}` | Retrieve order by ID (enriched) | 200, 404 |
| POST | `/api/orders` | Create a new order | 201, 400 |
| DELETE | `/api/orders/{id}` | Delete an order | 204, 404 |

**Contract Dependency:**
The Consumer's `UserDTO` class mirrors the Provider's `User` response schema. Any change to the Provider's response format that is not reflected in `UserDTO` will cause runtime failures. This is the exact dependency that contract testing protects.

**Enrichment Pattern:**
When the Consumer serves an order, it calls the Provider's User API to fetch user details and combines them into an `OrderResponse` object containing both order and user information.

### 6.5 Phase 5 — Contract Testing Implementation

**Objective:** Implement Spring Cloud Contract on both Provider and Consumer sides to enforce API contracts.

#### Provider Side Configuration

**Maven Dependencies Added:**
- `spring-cloud-contract-verifier` (test scope) — Reads contracts, generates tests
- `io.rest-assured:spring-mock-mvc` (test scope) — HTTP assertion library for MockMvc
- `spring-cloud-contract-maven-plugin` (build plugin) — Generates tests and stubs JAR

**Contract Files Created:**

| Contract | HTTP Method | Endpoint | Validates |
|----------|-----------|----------|-----------|
| `should_return_user_by_id.yml` | GET | `/api/users/1` | Single user response structure |
| `should_return_all_users.yml` | GET | `/api/users` | Array of users response |
| `should_create_a_new_user.yml` | POST | `/api/users` | User creation with 201 status |

**Base Test Class:**
`BaseContractTest.java` — Abstract class annotated with `@SpringBootTest`, initializes `RestAssuredMockMvc.standaloneSetup(userController)` in `@BeforeEach`. All auto-generated contract tests extend this class.

**Build Output:**
- `mvn clean install` → 6 tests pass (5 contract + 1 smoke)
- Contracts: `should_return_user_by_id`, `should_return_all_users`, `should_create_a_new_user`, `should_update_user_by_id` (AI-generated), `should_delete_user_by_id` (AI-generated)
- Stubs JAR generated and installed to local Maven repository (`~/.m2/repository`)

#### Consumer Side Configuration

**Maven Dependencies Added:**
- `spring-cloud-contract-stub-runner` (test scope) — Loads Provider stubs and starts WireMock

**Test Configuration:**
```java
@AutoConfigureStubRunner(
    ids = "com.contracttest:provider-api:+:stubs:8080",
    stubsMode = StubRunnerProperties.StubsMode.LOCAL
)
```

**Build Output:**
- `mvn clean test` → 2 tests pass (1 contract + 1 smoke)
- WireMock serves Provider stubs on port 8080 during test execution

### 6.6 Phase 6 — Breaking Change Verification

**Objective:** Demonstrate that contract testing catches breaking changes at build time.

**Test Procedure:**

| Step | Action | Expected Result |
|------|--------|-----------------|
| 1 | Modify contract: change `name` → `fullName` in `should_return_user_by_id.yml` | — |
| 2 | Run `mvn clean test` on Provider | **Build failure** |
| 3 | Observe error output | `JSON path [$.fullName] doesn't match. Expected: Alice Johnson, Actual: null` |
| 4 | Revert change | Tests pass again |

**Conclusion:**
The SCC Verifier detected that the Provider's actual API response (`name`) did not match the contract expectation (`fullName`). The build was blocked, preventing the breaking change from reaching deployable artifacts. This validates the contract testing mechanism is functioning correctly.

### 6.7 Phase 7 — AI Agent for Contract Generation

**Objective:** Build a Python-based AI agent that automates contract file generation by reading the Provider's OpenAPI specification.

**Implementation:**

The AI Agent lives in `ai-agent/` and consists of 5 tools orchestrated through a single CLI entry point (`main.py`).

| Tool | File | Purpose |
|------|------|---------|
| Spec Reader | `spec_reader.py` | Fetches `/v3/api-docs`, resolves `$ref` references, extracts all endpoints with schemas |
| Contract Generator | `contract_generator.py` | Converts parsed endpoints into SCC-compatible YAML contracts with matchers and sample data |
| Drift Detector | `drift_detector.py` | Compares existing contracts against live spec; finds uncovered, orphaned, and drifted contracts |
| Report Generator | `report_generator.py` | Produces formatted reports with coverage bars, health status, and remediation suggestions |
| CI Config Generator | `ci_config_generator.py` | Scans project structure, generates multi-stage `.gitlab-ci.yml` with caching and gated deploy |

**CLI Commands:**

| Command | Description |
|---------|-------------|
| `python main.py generate` | Fetch spec → generate contract YAMLs |
| `python main.py drift` | Compare contracts vs spec → report mismatches |
| `python main.py report` | Full health report with coverage and remediation |
| `python main.py validate` | CI-friendly check → exit code 0 (pass) or 1 (fail) |
| `python main.py ci` | Generate `.gitlab-ci.yml` pipeline |

**Results:**
- Generated **2 new contracts** (PUT update, DELETE) auto-verified by SCC
- **5/5 endpoints covered** — 100% HEALTHY
- Provider tests: **6/6 pass** (5 contract + 1 smoke)
- Consumer tests: **2/2 pass** (1 contract + 1 smoke)
- `.gitlab-ci.yml` generated with 7 jobs across 4 stages

**Test Isolation Fix:**
During development, the DELETE contract test was removing user 1, breaking subsequent GET tests. This was resolved by adding a `resetData()` method to `UserService` and calling it in `BaseContractTest.@BeforeEach`, ensuring each test starts with fresh sample data.

## 10. Key Terminology

| Term | Definition |
|------|-----------|
| **Contract** | A YAML file defining the expected request/response pair between a Consumer and Provider |
| **Provider** | The service that exposes an API (User Service in this project) |
| **Consumer** | The service that calls another service's API (Order Service in this project) |
| **Spring Cloud Contract (SCC)** | Framework for consumer-driven contract testing in the Spring ecosystem |
| **SCC Verifier** | Provider-side plugin that auto-generates JUnit tests from contract files |
| **SCC Stub Runner** | Consumer-side library that loads Provider stubs into a WireMock server |
| **WireMock** | HTTP mock server that serves pre-recorded responses for testing |
| **Stubs JAR** | Build artifact containing WireMock-compatible response mappings |
| **Contract Drift** | Divergence between the API's actual behavior and its contract definitions |
| **OpenAPI** | Machine-readable API specification standard (auto-generated from code) |
| **Matchers** | Flexible contract validations using regex patterns instead of exact values |
| **DTO** | Data Transfer Object — carries data between service boundaries |
